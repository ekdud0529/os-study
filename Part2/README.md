# 조예은
### 프로세스 메모리 배치에 대해서 설명해주세요
> 프로세스의 메모리는 4개의 섹션으로 구분됩니다. 텍스트 섹션에는 실행 코드, 데이터 섹션에는 전역 변수, 힙 섹션에는 프로그램 실행 중 동적으로 할당되는 메모리, 스택 섹션에는 함수를 호출할 때 필요한 임시 데이터를 저장합니다. 스택과 힙 섹션은 프로그램 실행 중에 동적으로 크기가 변합니다.
    
### 프로세스의 5가지 상태에 대해 설명해주세요
> - new : 생성 중인 프로세스
> - running : 명령어가 실행 중인 프로세스
> - waiting : 이벤트가 일어나기를 기다리는 프로세스
> - ready : 처리기에 할당되기를 기다리는 프로세스
> - terminated : 실행이 종료된 프로세스
    
###  문맥 교환에 대해 설명해주세요
> 인터럽트나 시스템 콜이 발생했을 때 CPU 코어를 다른 프로세스로 교환하기 위해 이전의 프로세스 상태를 PCB에 저장하고 새로운 프로세스의 상태를 복구하는 작업입니다.
    
###  고아 프로세스에 대해 설명해주세요
> 부모 프로세스가 wait()을 하지 않고 종료하면 그 자식 프로세스를 고아 프로세스라고 합니다. 이러한 경우 UNIX에서는 init 프로세스를 고아 프로세스의 부모 프로세스로 지정하고, init 프로세스는 주기적으로 wait()를 호출하여 고아 프로세스의 종료 상태를 수집하고 프로세스 식별자와 프로세스 테이블 항목을 반환합니다.
    
### 프로세스 간 통신 방법에 대해 설명해주세요
> 프로세스 간 통신 방법에는 공유 메모리를 사용하는 방식과 메시지 전달 방식이 있습니다. 공유 메모리 모델은 협력 프로세스들에 의해 공유되는 메모리 영역이 구축되고 프로세스들은 그 영역에 데이터를 읽고 씀으로써 정보를 교환합니다. 메시지 전달 모델에서는 협력 프로세스들 사이에 메시지를 교환하는 방식으로 통신이 이루어집니다.
    
###  파이프에 대해 설명해주세요
> 파이프는 두 프로세스가 통신할 수 있게 하는 전달자의 역할을 합니다. 일반 파이프는 생산자-소비자 형태로 생산자는 쓰기만 소비자는 읽기만 할 수 있습니다. 지명 파이프는 양방향 통신이 가능하고, 부모-자식 관계가 필요하지 않습니다. 또 통신 프로세스가 종료되더라도 계속 존재합니다.
    
###  소켓에 대해 설명해주세요
> 소켓은 클라이언트-서버 구조에서의 통신 방법입니다. 

###  스레드에 대해 설명해주세요
> 스레드는 CPU이용의 기본 단위로 스레드 ID, 프로그램 카운터, 레지스터 집합, 스택으로 구성됩니다. 스레드는 같은 프로세스에 속한 다른 스레드와 코드, 데이터 섹션, 열린 파일이나 신호같은 운영체제 자원을 공유합니다.
    
###  스레드 풀에 대해 설명해주세요
> 스레드 풀은 프로세스를 시작할 때 일한 수의 스레드를 미리 만들어 두는 것입니다.

# 김다영
### 프로세스란?
> 메모리에 적재된 프로그램으로 실행중인 프로그램이다.

### 프로세스 상태 5가지
> new
> running
> waiting
> ready
> terminated

### PCB가 무엇인가?
> 프로세스 상태, 프로그램 카운터 등 프로세스에 관한 정보가 수록된 곳이다.

### 프로세스 스케줄링의 목적은?
> 멀티프로그래밍과 시분할을 이루는 것이다.

### dispatch가 무엇인가?
> 프로세스가 ready 상태에서 running 상태가 되는 것

### Context Switch란?
> 인터럽트 발생 시, CPU 코어른 다른 프로세스 교환하기 위해서 이전의 프로세스 상태를 보관하고, 새로운 프로세스의 보관된 상태를 복구하는 작업이다.

### pid란?
> 프로세스 식별자다.

### 좀비 프로세스란?
> 자식 프로세스가 종료됐는데, 부모가 wait()를 호출하지 않은 프로세스

### 공유메모리 모델이란?
> 통신하는 프로세스들이 공유 메모리 영역을 구축해 데이터를 교환하는 모델

### 메시지 패싱 모델이란?
> 협력 프로세스 사이에서 메시지 교환을 통해 통신하는 모델

### 스레드란?
> CPU 이용의 기본단위로 프로세스의 작업 흐름이다.

### 싱글 스레드와 멀티 스레드 차이점
> 싱글은 한 번에 하나의 작업만 수행, 멀티는 동시에 여러 작업 수행 가능

### 멀티 스레드 장점 4가지
> 응답성
> 자원 공유
> 경제성
> 확장성

### 멀티 스레드에서 동시성과 병렬성이란?
> 싱글 코어에서 여러 스레드가 진행중인경우 동시성, 멀티 코어에서 여러 스레드가 동시에 진행중인 경우 병렬성이 존재하는 것이다.

### 다대일 모델이란?
> 많은 사용자 스레드를 하나의 커널 스레드가 대응하는 모델
### 일대일 모델
> 각 사용자 스레드를 각각 하나의 커널 스레드가 대응 하는 모델
### 다대다 모델
> 여러 개의 사용자 스레드를 그보다 작거나 같은 수의 커널 스레드가 대응하는 모델
### 두 수준 모델
> 다대다 모델과 ㅠ사한데, 한 사용 스레드가 하나의 커널 스레드에만 연관되는 것을 허용하는 모델

### 스레드 풀이란?
> 스레드를 요청할 때마다 새로운 스레드를 생성, 수행, 삭제를 통해 성능이 저하되는 것을 방지하기 위해 미리 스레드 풀에 여러개 스레드를 만들어두고, 요청이 오면 스레드 풀에서 할당해주는 방법

### CPU 스케줄러란?
> CPU가 유휴 상태일 때, 레디큐에 있는 어떤 프로세스에게 CPU 코어를 할당할지 결정한다.

### 비선점 스케줄링 방식이란?
> 프로세스가 자발적으로 CPU 제어를 포기하는 것

### 선점 스케줄링 방식이란?
> 프로세스가 CPU를 강제로 뺏을 수 있는 것

### 디스패치 지연이란?
> 디스패처가 하나의 프로세스를 정지하고, 다른 프로세스 수행을 시작하는 데가지 소요되는 시간

### 스케줄링 기준 4가지
> CPU 이용률
> 처리량
> 총 처리 시간
> 응답 시간

### FCFS 스케줄링?
> CPU를 먼저 요청하는 프로세스에게 CPU를 할당해주는 스케줄링

### SJF 스케줄링?
> 다음 프로세스 중 프로세스 수행 시간이 가장 짧은 프로세스에게 CPU를 할당하는 스케줄링

### RR 스케줄링?
> 레디큐를 돌며 한 프로세스에게 한 번의 시간할당량 동안 CPU를 할당하는 스케줄링

### 우선tns위 스케줄링?
> 특정 기준으로 프로세스에 우선순위를 부여해, 우선선위에 따라 CPU를 할당하는 알고리즘

### 다단계 큐 스케줄
> 프로세스 우선순위에 따라 여러 개의 큐로 분할하고, 가장 높은 우선순의 큐부터 프로세스를 스케줄하는 방식

### 다단계 피드백 큐
> 다단계 큐와 유사하지만, 다른 큐로 프로세스의 이동을 허용하는 방식

# 조민서
## 3장 
### 프로그램과 프로세스의 차이점에 대해 설명해 주세요.
>프로그램은 레지스터 명령어들이 파일에 저장된 수동적인 형태이고, 프로세스는 프로그램을 메인 메모리에 올려 프로그램 카운터와 관련 자원들을 가진 능동적인 형태입니다.


### 프로세스 스케줄링에 대해 설명해 주세요.
>프로세스가 시스템에 들어가면 먼저 준비큐로 들어가고 CPU 스케줄링에 의해 CPU 코어가 할당되기를 기다립니다. 
>
만약 프로세스에 `인터럽트 요청(예를 들어 입출력 I/O)`이 오면 프로세스는 대기큐에 들어가서 대기했다가 인터럽트 요청이 해결되면 다시 준비큐로 들어가 CPU 코어 할당을 기다립니다.

>#### 꼬리질문: 그럼 인터럽트 요청이 해결 된 것은 어떻게 알릴까요?
> 나: 인터럽트 요청하기 전에 프로세스 스택에 현재 주소를 저장하고 인터럽트 요청을 보냅니다. 그럼 인터럽트 서비스 루틴에서 함수를 실행 후 저장된 주소로 복귀합니다.

### CPU 스케줄링에 대해 설명해 주세요.
> CPU 스케줄링은 준비큐에 있는 프로세스 중에 어떤 프로세스에 CPU를 할당할지 결정하는 작업입니다. 

> CPU 스케줄링 방법으로는 프로세스에 우선순위를 둘 수도 있고, 준비큐에 먼저 온 프로세스부터 처리해도 되고, 프로세스 CPU 작업 시간이 짧은 순서 대로 처리를 하는 방법 등이 있습니다.

### fork()와 exec() 시스템 콜에 대해 설명해 주세요.
>  fork() 시스템 콜은 새로운 프로세스를 위한 메모리를 할당합니다. fork()를 호출하면 원래의 프로세스와 똑같은 코드를 가지는 프로세스를 생성하고, 새로운 프로세스는 fork()를 호출한 다음 라인부터 실행이 됩니다.

> 반면 exec() 시스템 콜은 fork()처럼 새로운 프로세스를 위한 메모리를 할당하지 않고, exec()를 호출한 프로세스가 아닌 exec()에 의해 호출된 프로세스만 메모리에 남게됩니다

> 
fork() = PID 가 완전히 다른 또 하나의 프로세스가 생기는 것 (새로운 PID)
exec() = 호출한 프로세스는 새로운 프로세스에 의해 덮어 쓰여지게 됩니다. (PID를 덮어 씌움)

### 프로세스끼리 통신하는 방법을 설명해 주세요.
>프로세스가 통신하는 방법에는 `공유 메모리 모델, 메시지 전달 모델`이 있습니다.

> `메모리 전달 모델`은 두 개 이상의 프로세스들이 주소 공간의 일부를 공유하며, 공유한 메모리 영역에 읽기/쓰기를 통해 통신을 수행합니다. (Read and Write)
>
>프로세스가 공유 메모리 할당을 커널에 요청하면, 커널은 해당 프로세스에 메모리 공간을 할당해 주게 되고, 이후 어떤 프로세스든지 해당 메모리 영역에 접근할 수 있다. 공유 메모리가 설정되면, 그 이후의 통신은 커널의 관여 없이 진행이 가능합니다.
>
> 커널의 관여 없이 메모리를 직접 사용하여 IPC 속도가 빠르고, 모든 프로세스끼리 자유로운 통신이 가능합니다. 하지만 메시지 전달 방식이 아니기에 데이터를 읽어야 하는 시점을 알 수 없어서 동기화 기술이 필요하고, 동시에 같은 메모리 위치를 접근하는 위치가 발생할 수 있습니다.

>`메시지 전달 모델`은 커널 메모리 영역에 메시지 전달을 위한 채널을 만들어서 협력하는 프로세스들 사이에 메시지 형태로 정보를 Send/Receive 하는 방법입니다. 커널을 경유하여 메시지를 송/수신자끼리 주고 받으며, 커널에서는 데이터를 버퍼링합니다.
>
>메시지 전달 모델은 메모리 공유 없이 동작하고, 커널에서 데이터들을 컨트롤 하므로 별도의 동기화 로직이 없어도 되지만 커널을 통해서 데이터를 주고 받기 때문에 공유 메모리 모델보다 속도가 느립니다.
>
> 메시지 전달 모델 예시) 파이프, 메시지큐, 소켓

### 파이프에 대해 설명해 주세요.
> 파이프 종류에는 `일반 파이프와 지명 파이프`가 있습니다.
>
>`일반 파이프`는 프로세스 두 개를 연결하여 하나의 프로세스는 데이터를 쓰기만 하고 다른 프로세스는 데이터를 읽기만 하는 단순한 데이터 흐름입니다. 주로 부모-자식 간의 단방향 통신으로 사용합니다.
>
> `지명 파이프`는 양방향 통신이 가능하며, 부모-자식 관계도 필요로 하지 않습니다. 하지만 일반 파이프, 지명 파이프 모두 같은 기기 내에서만 통신할 수 있습니다.

### 소켓에 대해 설명해 주세요.
>파이프와 지명 파이프로는 다른 기계에 있는 프로세스와 통신할 수 없습니다. 그래서 다른 기계와 통신하기 위해 사용되는 것이 소켓이다. 소켓이란 통신의 극점을 의미하며, 각 소켓은 IP주소와 포트 번호를 가집니다.

### 원격 프로시저 호출(Remote Procedure Calls, PRC)에 대해 설명해 주세요.
> 원격 프로시저 호출은 다른 주소 공간에서 함수나 프로시저를 실행할 수 있게하는 프로세스 간 통신 기술입니다. 

> 원격 프로시저 호출을 이용하면 프로그래머는 함수가 실행 프로그램에 로컬 위치에 있든 원격 위치에 있든 동일한 코드를 이용할 수 있습니다.

>#### PRC 필요한 이유 
MSA(Micro Service Architecture) 구조로 서비스를 만든다면, 다양한 언어와 프레임워크로 개발이 되는데, 이런 구조에서는 프로토콜을 맞춰서 통신해야 하는 비용이 발생한다.
이 경우에 RPC를 이용하여 언어에 구애받지 않고, 원격 프로시저를 호출하여 비즈니스 로직에 집중할 수 있습니다.

![](https://velog.velcdn.com/images/minseojo/post/2a10527f-a9c1-4265-818d-9fdb4782e1e0/image.png)

1. 클라이언트가 일반적인 방식으로 파라미터를 넘겨 `클라이언트스텁 프로시저`를 호출한다. 클라이언트스텁은 클라이언트를 소유한 주소의 공간 내에 거주한다.

2. `클라이언트스텁`이 파라미터들을 메세지로 모은다. 여기서 모은다는 것에 파라미터의 표현을 표준 포맷으로 변경하고 각 파라미터를 복사해서 메세지로 넣는 것도 포함된다.

3. `클라이언트스텁`은 원격 서버 머신으로 메세지를 보내는 계층인 `전송계층으로` 메세지를 보낸다.

4. 서버에서, `전송계층은` 메세지를 `서버스텁으로` 보낸다. 서버스텁은 또 파라미터들을 모아주고 일반적인 프로시저 호출 메커니즘을 사용하여 요구된 서버 루틴을 호출한다.

5. 서버 프로시저가 완료될 때, `서버 프로시저는 서버스텁으로 반환된다.` (이를테면 일반적인 프로시저 호출 반환값을 통해), `서버스텁은 결과 값들을 모아서 메세지에 넣고, 전송계층에 메세지를 보낸다.`

6. `전송계층`은 결과 메세지를 다시 클라이언트 전송계층으로 보내고 클라이언트 전송계층 그 결과를 클라이언트스텁에게 전달한다.

7. 클라이언트스텁은 반환 파라미터들과 실행 결과값을 다시 해체한다.

>#### 꼬리질문: PCR이 언어에 구애받지 않는다고 했는데 그럼 스텁은 어떤 식으로 구현되어 있나요? 예를 들어 자바랑 c언어는 데이터들마다 메모리 저장 방식이 다른데 이 데이터들을 어떻게 구별하고 처리하죠? 
>나: IDL(Interface Definition Language)을 사용하여 서버의 호출 규약을 정의하고, 스텁코드는 데이터형을 XDR(External Data Representation) 형식으로 변환하여 RPC 호출을 실행합니다.
